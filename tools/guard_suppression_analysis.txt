## Architecture Overview

The guard suppression during attacks involves THREE layers:

### Layer 1: StageChart InputBlock (Data-Driven)
- Class: StageChart_Function_InputBlock
- vtable: 0x144AFCC70
- Key function: slot[3] at 0x141C8F760 (696 bytes)
- This function iterates over InputBlock entries in the .paac stage chart data
- Each InputBlock entry specifies an "input group" to block
- Returns non-zero block code if the input's group is blocked
- Returns 0 if input is allowed

### Layer 2: InputBlockFromCode (Code-Driven)
- Function: 0x14319E1F0
- Called from: 0x140AD5001 (inside combat action setup at 0x140AD4F00)
- Creates an InputBlock at object offset +0x31190
- Only called when global flag at [rip+0x517432A] (VA ~0x145C4931A) is non-zero
- Takes dl=1 to enable, dl=0 to disable

### Layer 3: Themida-Protected Input Validator
- Function 0x140745E90 -> jumps to 0x148CBAB20 (Themida-encrypted CODE section)
- This is called BEFORE the detailed input check at 0x140745330
- Located in CODE section at VA 0x147BFB000, size 0x1350000
- Cannot be analyzed without runtime decryption (VEH + NtAllocateVirtualMemory trick)

## Key Functions

### 0x140AD4F00 - Combat Action Entry Gate
- Args: rcx=player_obj, dx=action_id(u16)
- Flow:
  1. Check byte [rcx+0x9C8] override flag
  2. If NOT set: call 0x140745E90(rsi) [Themida] -> if true, allow
  3. If NOT set and Themida returns false: call 0x140745330(rsi, bx) [unencrypted]
  4. If returns true -> proceed to execute action
  5. Sets [rdi+0x9C0] = action_id
  6. Sets byte [rdi+0x9CA] = 1 (action active)
  7. Calls [rdi+0x558]->vtable[5] (sequencer start)
  8. Optionally calls InputBlockFromCode to block further input

### 0x140745330 - Detailed Input Permission Check (~1400 bytes, UNENCRYPTED)
- Args: rcx=input_component, dx=action_id(u16)
- Checks a global enable flag at [rip+0x5503FAE]
- Checks for 0xFFFF (invalid action sentinel)
- Calls 0x1406B0BE0 to get sequencer state
- Calls 0x140746F40 to check if action is valid in current state
- Checks current sequencer action vs requested action
- Compares action against slot data at [r8+0xCA] and [r8+0xD2]
- This is the function that decides if guard (or any input) can proceed

### 0x140746F40 - Action Validity in Current State
- Checks [action_data+0x68]->word[0x30] against the requested action
- Calls 0x1412F4830 to check if action is valid during current sequence
- Checks [action_data+0x68]+0x190 -> iterates a 3-entry array checking action slots
- Calls 0x140747120 at end to process the transition

## Approach for Guard Cancel Mod

### Option A: Patch InputBlock Data (PREFERRED if feasible)
The InputBlock system reads data from the .paac stage chart. If guard is blocked via
an InputBlock entry in the attack state's chart, removing/modifying that entry would
allow guard during attacks. This is a DATA patch, no code modification needed.
- Need to identify which _inputBlockType value corresponds to guard
- Need to find the InputBlock entries in twohandsword attack states

### Option B: Patch 0x140745330 (Unencrypted Input Permission Check)
This ~1400 byte function decides if actions are valid in the current state.
Could be patched to always allow guard actions regardless of state.
- Would need to identify the guard action ID (u16 value)
- Would need to NOP or bypass the rejection path for that specific action
- Risk: may cause issues with other input blocking logic

### Option C: Patch InputBlockFromCode caller (0x140AD4FE7)
The byte at [rip+0x517432A] (VA ~0x145C4931A) controls whether InputBlockFromCode
is called. Setting this to 0 would disable code-driven input blocks entirely.
- AOB: 80 3D ?? ?? ?? ?? 00 74 16 at 0x140AD4FE7
- Patch: Change 74 16 (je skip) to EB 16 (jmp skip) to ALWAYS skip InputBlockFromCode
- Risk: this only affects code-driven blocks, not data-driven StageChart blocks

### Option D: Hook input dispatch at runtime (ASI mod)
Similar to HurryTheFUp approach:
1. AOB scan for the InputBlock::slot[3] return check pattern
2. Hook to force return 0 (not blocked) when the input is guard
3. Needs CRC bypass for Themida protection if patching .sdata
